Every prediction market has the same hard problem: **who decides the result?**
Polymarket answers with people — a proposer, a dispute window, UMA voters. That is
trust-*minimized*, not trust-*less*. For an objective sports outcome ("did Brazil
advance?"), a math-proven settlement is strictly better than a vote.

Bracket Bond settles on a **cryptographic proof of the actual match data**, anchored
on Solana by TxLINE, and enforced inside the program.

## The trust rule

<Callout type="info">
  One rule decides where each data point comes from: **does money or the official
  result depend on it?** Yes → TxLINE only. No → any API is fine. Odds, scores, and
  the settlement proof are TxLINE; flags and cosmetics can be anything.
</Callout>

Markets are therefore limited to **objective, TxLINE-provable outcomes** —
advancement, goals, cards, corners, shootouts. A subjective market ("was that VAR
call correct?") would need an oracle and would dissolve the differentiator.

## How it works

TxLINE hashes every scores update into a Merkle tree and publishes a **daily root**
to Solana (a PDA under the `Txoracle` program). Anyone can later prove a specific
stat was published and unaltered.

<Mermaid chart={`

  participant K as Keeper
  participant T as TxLINE API
  participant BB as bracket_bond
  participant TX as Txoracle (Solana)
  K->>T: GET /scores/stat-validation (fixture, seq, statKeys=1,2)
  T-->>K: stat + Merkle proof
  K->>BB: settle_round(eliminate=loser, proof)
  BB->>TX: CPI validate_stat(payload, strategy)
  TX-->>BB: return bool (proof verified vs daily root)
  BB->>BB: require bool == true, else revert (ProofFailed)
  BB->>BB: mark loser Eliminated · forfeit collateral to the pot
`} />

The program does not trust the keeper. `settle_round` **relays the validate call
as a CPI and reads its return value** — settlement applies only if the proof
verifies:

```rust
invoke(&validate_ix, &accounts)?;                 // Txoracle.validate_stat
let (prog, data) = get_return_data().ok_or(ProofFailed)?;
require_keys_eq!(prog, config.txoracle_program, BadOracleProgram);
require!(data.first() == Some(&1u8), ProofFailed); // bool must be true
```

## The advancement predicate (including shootouts)

"Did participant A advance?" needs care. TxLINE's full-game goal stats (keys
`1`/`2`) **exclude penalty-shootout goals**, so a knockout decided on penalties is
*level* at full time — a naive `goals A − B > 0` would settle it as a draw. Two
paths:

- **Regulation / extra time:** `Binary(goalsA − goalsB > 0)`.
- **Level at full time → penalties:** `Binary(goalsA − goalsB == 0)` **AND**
  `Binary(peA − peB > 0)`, using the penalty keys **6001/6002** (the live feed
  shifts the documented period table — `+5000` is actually ET2; PE is `+6000`).

`determineAdvancement` (`scripts/txline/statValidation.ts`) runs this with the
proof itself: it checks "level after full time"; if not, it proves the regulation
winner; if level, it fetches the PE stats and proves the shootout winner — then
returns the winning `validateStatV2` instruction to relay through `settle_round`.
No score-field parsing, no trusted input.

<Callout type="info">
  Most other submissions punt on shootouts — a QF/SF decided on penalties reads as
  a draw under goals-only predicates. Settling knockout ties correctly (and
  proving a real shootout on devnet) is our moat.
</Callout>

## Modes

| `settlement_mode` | Behaviour | Use |
| --- | --- | --- |
| `PROOF` | `settle_round` requires a `validate_stat` CPI to return `true` | Production, live TxLINE |
| `TRUSTED_ORACLE` | The oracle authority signs the result directly | Local tests, the replay demo |

The same `settle_round` code path runs the proof CPI in `PROOF` mode; the mode is
a config flag, not a fork in the logic.

## Trustless binding

`settle_round` doesn't just require the proof to return `true` — it also **binds
the proof to this outcome's fixture on-chain**. Each `Outcome` stores an
`expected_fixture_id`; `settle_round` parses the fixture id out of the relayed
`validateStatV2` data and requires it to match before eliminating. So an oracle
can't prove match A and eliminate a team from match B — the fixture is pinned, and
every settlement emits a `RoundSettled` event for the live feed.

The next hardening (following ShroudLine's model): build the predicate strategy
on-chain and pin each stat key to a fixed index, so the *predicate* — not just the
fixture — is program-enforced. Tracked in the roadmap.
